# 项目开发顶层规范提示词

> 适用目标：本地可开发、可测试、可备份；服务器可部署、可重启、可迁移；后续可多人使用、可持续维护。

## 一、总目标

请按“可部署、可分发、可维护”的工业化方式开发项目。不要只做能在本机临时跑通的 Demo。项目从第一天开始就要满足：

- 本地开发和服务器生产环境隔离。
- 不写死本地绝对路径。
- 用户上传、导出、数据库、日志都有统一目录和配置。
- 支持账号体系、权限隔离、操作日志、数据备份。
- 支持部署到 Linux ECS/云服务器。
- 支持 HTTPS 反向代理，避免浏览器阻止不安全下载。

## 二、项目结构规范

项目根目录必须清晰，推荐结构：

```text
project/
  app/
    main.py
    config.py
    database.py
    models.py
    security.py
    services/
    templates/
    static/
  data/
    uploads/
    exports/
    backups/
  logs/
  scripts/
  tests/
  requirements.txt
  .env.example
  README.md
  Dockerfile
  docker-compose.yml
```

要求：

- 所有路径都从项目根目录或环境变量解析。
- 禁止在代码里写死 `C:\Users\...`、`/root/...` 等本地绝对路径。
- `data/`、`logs/` 必须可配置。
- 上传文件和导出文件必须分目录存储。
- 输出文件名必须唯一，避免并发覆盖。

## 三、环境配置规范

必须提供 `.env.example`，生产环境用 `.env` 覆盖。

必须至少包含：

```env
APP_ENV=development
APP_NAME=分销记账工具
SECRET_KEY=change-me
DATABASE_URL=sqlite:///data/app.db
STORAGE_DIR=data
UPLOAD_DIR=data/uploads
EXPORT_DIR=data/exports
LOG_LEVEL=INFO
GRANDMASTER_USERNAME=admin
GRANDMASTER_PASSWORD=ChangeMe123!
```

要求：

- 开发环境和生产环境通过 `APP_ENV` 区分。
- 密钥、数据库地址、管理员初始账号都走配置。
- 生产环境不能依赖本机路径。
- 服务器部署时必须明确 `PYTHONPATH` 或使用 `python -m uvicorn app.main:app`。

## 四、依赖和版本规范

必须固定依赖版本，避免服务器上装到不兼容版本。

`requirements.txt` 不要只写包名，建议写成：

```txt
fastapi==0.x.x
uvicorn[standard]==0.x.x
sqlalchemy==2.x.x
jinja2==3.x.x
starlette==0.x.x
python-multipart==0.x.x
openpyxl==3.x.x
python-docx==1.x.x
itsdangerous==2.x.x
```

要求：

- 本地和服务器使用同一份 `requirements.txt`。
- 不允许服务器上“缺什么补什么”。
- 发现 Jinja2、Starlette、FastAPI API 不兼容时，优先锁版本，不靠临时改代码硬适配。

## 五、启动和部署规范

本地启动：

```bash
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

服务器启动：

```bash
cd /path/to/project
export PYTHONPATH=/path/to/project
venv/bin/python -m uvicorn app.main:app --host 0.0.0.0 --port 8888 --workers 2
```

生产环境必须使用 `systemd` 或进程管理工具，不依赖 SSH 会话：

```ini
[Unit]
Description=Distributor Ledger App
After=network.target

[Service]
WorkingDirectory=/path/to/project
Environment=PYTHONPATH=/path/to/project
EnvironmentFile=/path/to/project/.env
ExecStart=/path/to/project/venv/bin/python -m uvicorn app.main:app --host 127.0.0.1 --port 8888 --workers 2
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

要求：

- 不建议直接暴露 uvicorn 到公网。
- 推荐 Nginx 监听 80/443，反代到 `127.0.0.1:8888`。
- 生产下载、页面、静态资源必须使用同一个 HTTPS 域名。

## 六、HTTPS 和下载安全规范

浏览器阻止“不安全下载”的常见原因：

- 页面是 HTTPS，但下载链接是 HTTP。
- 文件下载跳转到 IP + 端口。
- MIME 类型不正确。
- 下载接口没有 `Content-Disposition: attachment`。

要求：

- 正式部署必须配置 HTTPS。
- 页面和 `/files/{id}` 下载接口必须同域同协议。
- 不能出现 `https://domain/page` 下载 `http://ip:8888/files/1`。
- 文件响应必须包含正确 MIME：
  - `.xlsx`: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
  - `.docx`: `application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- 文件响应建议包含：

```http
Content-Disposition: attachment
X-Content-Type-Options: nosniff
Cache-Control: private, no-store
```

## 七、账号和权限规范

系统至少有两类用户：

- 总舵主：唯一超级管理员，同时也可以拥有自己的人员数据。
- 分销商：只能访问和操作自己名下数据。

总舵主有双重身份：

1. 超级管理员身份：查看所有分销商数据、管理邀请码、管理账号、查看日志。
2. 业务身份：可以新增和管理“总舵组”自己的人员，核准也只能核准总舵组人员。

权限要求：

- 分销商只能看到自己的人员、导入、导出、核准。
- 总舵主可以查看全部分组数据。
- 总舵主不能代替分销商核准分销商人员。
- 总舵主“我的”页面的佣金，只计算总舵组自己的人员。
- 总舵主主页可显示：
  - 总舵应返
  - 分销商应返
  - 分销商数量
  - 各分销商核准进度

## 八、注册和邀请码规范

分销商注册必须使用总舵主生成的邀请码。

邀请码要求：

- 总舵主可新建邀请码。
- 总舵主可作废邀请码。
- 邀请码可以多人使用。
- 每个分销商账号必须记录自己使用的邀请码。
- 总舵主可以看到某个邀请码被哪些分销商使用、使用时间。
- 邀请码备注属于内部字段，只给总舵主看。

建议表设计：

```text
invitation_codes:
  id
  code
  created_by_id
  is_active
  note
  created_at
  deleted_at

users:
  id
  username
  display_name
  password_hash
  role
  invitation_code
  is_active
  created_at
```

## 九、密码和安全规范

要求：

- 密码不能明文存储。
- 使用 PBKDF2、bcrypt 或 Argon2 存储哈希。
- 用户必须允许修改自己的密码。
- 修改密码必须校验当前密码。
- 新密码至少 8 位。
- 修改后立即写入数据库，服务器重启后仍有效。
- 关键操作写入审计日志。

## 十、业务数据模型规范

人员资料是长期资料，不应简单等同某个月的一行数据。

人员统计按“截至某月快照”计算：

- 人员有起始月份。
- 起始月份优先取入职时间。
- 如果没有入职时间，则取实际创建/导入月份。
- 不应把 5 月导入的数据自动补到 1-4 月。
- 同一个人按 `SFid -> 残疾证ID -> 姓名` 逐级去重。
- 查看某个月时，只统计起始月份 <= 当前月份的人员。

金额统计：

- 只统计在职人员。
- 离职人员只进入总资料和离职人数，不参与金额和核准。
- 总舵主自己的佣金只算总舵组人员。
- 分销商佣金只算该分销商人员。

## 十一、月度核准规范

核准是月度行为，不能直接挂在人员资料上。

必须设计独立月度核准表：

```text
monthly_confirmations:
  id
  person_id
  period
  confirmed_by_id
  confirmed_at
  unique(person_id, period)
```

要求：

- 2026-05 核准不能影响 2026-06。
- 总舵主只能核准总舵组人员。
- 分销商只能核准自己的人员。
- 总舵主可以查看分销商核准进度，例如 `16/18`。

## 十二、导入和导出规范

上传：

- 用户文件只能通过上传接口进入系统。
- 禁止要求用户填写服务器本地路径。
- 上传文件保存到 `UPLOAD_DIR`。
- 上传文件名必须唯一。
- 上传失败必须记录日志。

导出：

- 导出文件保存到 `EXPORT_DIR`。
- 导出文件名必须唯一。
- 生成文件后先写数据库记录，再提供下载。
- 导出中心应显示顶部下载入口。
- 生成后可自动触发一次下载。
- 提供“查看服务器路径”辅助调试，但用户真正下载应通过浏览器。

## 十三、日志和审计规范

必须有两类日志：

1. 程序日志：写入 `logs/app.log`。
2. 业务审计日志：写入数据库。

审计日志至少记录：

- 注册
- 登录失败可选
- 修改密码
- 新增人员
- 编辑人员
- 删除人员
- 导入人员
- 导出文件
- 月度核准
- 新建邀请码
- 作废邀请码
- 修改分销商信息

审计日志页面应紧凑、可折叠，不要每条大卡片。

## 十四、数据库初始化和迁移规范

要求：

- 首次启动自动创建表。
- 如果使用 SQLite，必须确保 `data/` 目录存在。
- 总舵主账号必须自动初始化。
- 如果总舵主密码哈希为空，启动时应修复或报警。
- 后续新增字段必须有兼容旧库的迁移逻辑。
- 不允许生产环境直接删库解决问题，必须先备份。

清理测试数据必须：

- 人工确认。
- 先备份数据库到 `data/backups/`。
- 说明保留哪些账号、删除哪些数据。

## 十五、前端交互规范

应用面向手机移动端使用，要求：

- 底部导航：主页、新增、导入、我的；总舵主额外有管理。
- 月份选择使用 1-12 月快捷按钮，不使用繁琐的年月输入 + 切换按钮。
- 多用户/多分组场景默认折叠，避免页面臃肿。
- 总舵主看到分组：总舵组、分销商 A、分销商 B。
- 分销商只看到自己的组。
- 导出生成后，下载入口出现在顶部提示附近。
- 错误反馈显示在操作区域附近。

## 十六、部署前检查清单

上线前必须检查：

- [ ] `requirements.txt` 锁定版本。
- [ ] `.env.example` 完整。
- [ ] 生产 `.env` 已配置。
- [ ] `SECRET_KEY` 已更换。
- [ ] 总舵主默认密码已修改。
- [ ] 数据库路径在服务器可写。
- [ ] `data/uploads`、`data/exports`、`logs` 目录存在。
- [ ] Nginx HTTPS 已配置。
- [ ] `/files/{id}` 下载不跳转 HTTP。
- [ ] systemd 服务可自动重启。
- [ ] 数据库备份策略已配置。
- [ ] 防火墙/安全组只开放必要端口。

## 十七、交付要求

每次开发完成必须给出：

- 修改了哪些功能。
- 涉及哪些文件。
- 如何启动。
- 如何测试。
- 是否重启服务。
- 是否涉及数据库迁移。
- 是否需要备份。
- 生产部署注意事项。

## 十八、给 AI/开发者的执行原则

开发时请遵守：

- 不要只让功能在当前电脑临时跑通。
- 先确认数据流：输入、校验、存储、处理、输出。
- 权限判断必须在后端做，不能只靠前端隐藏按钮。
- 文件下载、上传、数据库写入都要有异常处理和日志。
- 涉及删除或清理数据，必须先备份并人工确认。
- 涉及服务器部署，优先考虑 HTTPS、systemd、Nginx、备份。
- 不要把本地路径、账号密码、服务器 IP 写死进代码。
- 所有新增字段要考虑旧数据库升级。
