# 通用项目开发顶层规范提示词

> 用途：在任何 Web 应用、管理系统、SaaS 工具、内部业务系统开始开发前，作为顶层开发约束交给 AI 或开发团队使用。

## 一、总体目标

请按“可部署、可分发、可维护、可迁移”的工业化方式开发项目。不要只做能在当前电脑临时运行的 Demo。项目从第一天开始就要满足：

- 本地开发、测试环境、生产环境隔离。
- 不写死本地绝对路径、服务器 IP、端口、账号、密码、业务名称。
- 配置外置，所有环境差异通过 `.env` 或部署平台配置注入。
- 用户文件只能通过上传接口进入系统。
- 输出文件、缓存文件、日志文件、备份文件必须统一管理。
- 支持多用户、权限隔离、审计日志、数据备份。
- 支持部署到 Linux 服务器、云服务器、容器或 PaaS。
- 支持 HTTPS、反向代理、进程托管、自动重启。

## 二、项目结构规范

推荐结构：

```text
project/
  app/
    main.py
    config.py
    database.py
    models.py
    security.py
    services/
    templates/
    static/
  data/
    uploads/
    exports/
    backups/
  logs/
  scripts/
  tests/
  requirements.txt
  .env.example
  README.md
  Dockerfile
  docker-compose.yml
```

要求：

- 所有路径从项目根目录或环境变量解析。
- 禁止写死 `C:\...`、`/root/...`、`/home/...` 等绝对路径。
- 禁止在代码里写死服务器 IP、域名、端口。
- `data/`、`logs/`、上传目录、导出目录都必须可配置。
- 目录不存在时，应用启动或初始化脚本应安全创建。
- 临时文件、上传文件、导出文件必须分开存储。

## 三、环境配置规范

必须提供 `.env.example`，生产环境用 `.env` 或平台环境变量覆盖。

示例：

```env
APP_ENV=development
APP_NAME=Your App Name
SECRET_KEY=change-me
DATABASE_URL=sqlite:///data/app.db
STORAGE_DIR=data
UPLOAD_DIR=data/uploads
EXPORT_DIR=data/exports
LOG_DIR=logs
LOG_LEVEL=INFO
SERVER_HOST=127.0.0.1
SERVER_PORT=8000
PUBLIC_BASE_URL=http://127.0.0.1:8000
INITIAL_ADMIN_USERNAME=admin
INITIAL_ADMIN_PASSWORD=change-me-now
```

要求：

- 开发、测试、生产环境必须区分。
- 密钥、数据库地址、文件目录、公开访问域名都走配置。
- 生产环境不得使用默认 `SECRET_KEY`。
- 生产环境初始管理员密码必须强制修改或明确提示。
- 应用内部生成链接时，应优先使用 `PUBLIC_BASE_URL` 或请求上下文，不得写死地址。

## 四、依赖和版本规范

必须固定依赖版本，避免本地和服务器安装到不同版本。

要求：

- 使用 `requirements.txt`、`pyproject.toml`、`package-lock.json`、`pnpm-lock.yaml` 等锁定依赖。
- 不允许服务器上“缺什么补什么”。
- 本地、CI、服务器使用同一份依赖定义。
- 框架、模板引擎、ORM、ASGI/WSGI 服务器之间必须确认版本兼容。
- 发现版本兼容问题，优先锁定版本，而不是在服务器上临时改代码。

## 五、启动和部署规范

应用启动命令必须可配置，不固定端口。

本地示例：

```bash
python -m uvicorn app.main:app --host ${SERVER_HOST:-127.0.0.1} --port ${SERVER_PORT:-8000}
```

生产建议：

- 使用 systemd、Supervisor、Docker、Kubernetes 或云平台进程管理。
- 不依赖 SSH 会话保持进程。
- 不建议将应用服务器直接暴露公网。
- 推荐使用反向代理监听公开端口，再转发到应用内部监听地址。

反向代理原则：

```text
用户浏览器
  -> HTTPS 公网域名
  -> 反向代理
  -> 应用内部地址
```

注意：

- 反向代理监听端口、应用内部端口都必须通过配置决定。
- 文档中可以给示例端口，但不能作为硬编码要求。
- 生产部署必须说明如何重启、查看日志、回滚。

## 六、HTTPS 和下载安全规范

现代浏览器会拦截不安全下载。常见原因：

- HTTPS 页面下载 HTTP 文件。
- 下载链接跳到 IP + 明文端口。
- 下载接口和页面不是同域同协议。
- 文件 MIME 类型错误。
- 响应头缺失或内容被浏览器误判。

要求：

- 正式部署必须使用 HTTPS。
- 页面、静态资源、下载接口应使用同一个 HTTPS 域名。
- 不得让 HTTPS 页面下载 HTTP 文件。
- 文件下载必须通过后端授权接口，不直接暴露服务器文件路径。
- 文件响应必须包含正确的 `Content-Type`。
- 文件响应建议包含：

```http
Content-Disposition: attachment
X-Content-Type-Options: nosniff
Cache-Control: private, no-store
```

## 七、账号、角色和权限规范

系统如有用户体系，必须从一开始设计权限边界。

要求：

- 至少区分普通用户和管理员。
- 如存在多租户、团队、组织、渠道、部门等概念，必须做数据隔离。
- 权限判断必须在后端执行，不能只靠前端隐藏按钮。
- 管理员可以查看或管理哪些数据，必须明确。
- 普通用户只能访问自己被授权的数据。
- 一个账号如同时有“管理身份”和“业务身份”，必须分别计算权限和业务数据，不能混用。

示例原则：

```text
管理权限：查看、配置、审计、禁用账号。
业务权限：新增、编辑、核准、结算、导出自己的业务数据。
```

## 八、注册、邀请和账号管理规范

如果系统不允许开放注册，应使用邀请机制。

邀请机制要求：

- 管理员可生成邀请码或邀请链接。
- 管理员可作废邀请码。
- 是否允许一个邀请码多人使用，必须配置化或在需求中明确。
- 每个注册用户必须记录注册来源，例如邀请码、邀请人、注册时间。
- 管理员能查看邀请使用记录。
- 内部备注字段只给管理员看。

账号管理要求：

- 用户可以修改自己的密码。
- 管理员可以禁用账号。
- 密码不得明文存储。
- 修改密码必须校验当前密码。
- 密码修改后立即写入数据库，服务重启后仍有效。

## 九、数据模型和业务口径规范

开发前必须明确：

- 哪些是长期资料。
- 哪些是月度/周期性记录。
- 哪些是临时计算结果。
- 哪些需要历史快照。
- 哪些字段参与唯一性判断。

要求：

- 不要把长期资料和月度状态混在同一字段里。
- 周期性确认、结算、审批应使用独立记录表。
- 统计口径必须明确，例如按创建时间、开始时间、结束时间、状态、归属用户计算。
- 删除策略必须明确：物理删除、软删除、归档删除。
- 涉及金额、人数、状态的统计，必须统一从服务层函数计算，避免各页面各算各的。

## 十、文件上传和导出规范

上传要求：

- 用户文件只能通过上传接口进入系统。
- 禁止依赖用户本地路径。
- 上传文件必须校验扩展名、大小、内容类型。
- 上传文件名必须唯一。
- 上传失败必须记录日志。
- 上传目录必须可配置。

导出要求：

- 导出文件保存到配置化目录。
- 导出文件名必须唯一。
- 生成文件后写入数据库记录。
- 下载通过授权接口完成。
- 下载按钮应靠近生成成功提示。
- 生成耗时较长时，预留异步任务或队列扩展点。
- 不要直接暴露服务器真实文件路径给普通用户；可为管理员提供调试路径。

## 十一、日志和审计规范

必须区分：

1. 程序运行日志。
2. 用户操作审计日志。

程序日志：

- 写入配置化日志目录。
- 包含异常堆栈。
- 包含启动、停止、导入、导出、任务执行等关键事件。

审计日志：

- 写入数据库。
- 记录操作者、动作、目标类型、目标 ID、时间、说明。
- 至少记录：注册、登录失败可选、修改密码、新增、编辑、删除、导入、导出、审批、配置变更。
- 审计日志页面应支持筛选、折叠或分页，避免页面臃肿。

## 十二、数据库初始化、迁移和备份规范

要求：

- 首次启动自动创建必要表。
- 旧数据库新增字段时要有兼容迁移。
- 初始化管理员账号时，密码哈希不能为空。
- 数据库目录必须可写。
- 清理数据前必须人工确认。
- 清理或迁移前必须先备份。
- 备份文件应带时间戳。

备份目录示例：

```text
data/backups/app_before_migration_YYYYMMDD_HHMMSS.db
```

## 十三、前端交互规范

要求：

- 面向目标用户设计，不做无意义的展示页。
- 移动端优先时，常用操作必须在首屏或底部导航可达。
- 多用户、多团队、多分组数据默认折叠，避免页面臃肿。
- 操作反馈要出现在用户刚操作的位置附近。
- 导出成功后，下载入口要靠近成功提示。
- 错误提示要清楚说明下一步怎么做。
- 不用“已处理但没反馈”的交互。

## 十四、异步任务扩展规范

对于耗时任务，必须预留扩展点：

- 大文件导入。
- 大批量导出。
- 批量计算。
- 第三方接口同步。
- 定时统计。

即使第一版同步执行，也要让服务层可替换为任务队列：

```text
接口层 -> 任务创建 -> 服务层处理 -> 文件/结果存储 -> 状态查询 -> 下载/查看
```

## 十五、部署文档规范

README 或部署文档必须包含：

- 项目简介。
- 本地启动步骤。
- 环境变量说明。
- 数据库初始化方式。
- 依赖安装方式。
- 生产启动方式。
- 反向代理示例。
- HTTPS 注意事项。
- 文件上传/导出目录说明。
- 日志位置。
- 备份和恢复方法。
- 常见故障排查。

## 十六、上线前检查清单

上线前必须确认：

- [ ] 依赖版本已锁定。
- [ ] `.env.example` 完整。
- [ ] 生产环境变量已配置。
- [ ] `SECRET_KEY` 已更换。
- [ ] 初始管理员密码已修改。
- [ ] 数据库目录可写。
- [ ] 上传、导出、备份、日志目录存在且可写。
- [ ] 反向代理配置正确。
- [ ] HTTPS 可用。
- [ ] 下载接口不降级到 HTTP。
- [ ] 服务由 systemd、容器或平台托管。
- [ ] 服务异常会自动重启。
- [ ] 数据库有备份策略。
- [ ] 防火墙或安全组只开放必要端口。
- [ ] 权限隔离已测试。
- [ ] 导入、导出、下载已测试。

## 十七、交付说明规范

每次开发完成，必须说明：

- 修改了什么。
- 涉及哪些文件。
- 是否涉及数据库迁移。
- 是否需要重启服务。
- 如何本地测试。
- 如何服务器部署。
- 有哪些剩余风险。
- 是否需要用户人工确认。

## 十八、给 AI 或开发者的执行原则

请始终遵守：

- 先分析需求和角色边界，再写代码。
- 先设计数据流：输入、校验、存储、处理、输出。
- 不把业务规则散落在模板里。
- 不把权限规则只写在前端。
- 不把路径、端口、域名、账号密码写死。
- 不用删除数据库解决业务问题，除非用户明确确认且已备份。
- 所有 I/O 都要有异常处理和日志。
- 所有输出文件名必须唯一。
- 所有生产部署问题都要回到配置、版本、路径、权限、HTTPS、进程托管这六件事检查。
